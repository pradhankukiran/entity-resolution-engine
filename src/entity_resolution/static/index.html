System Overview
Architecture and methodology of the entity resolution pipeline.

What this system does

This engine resolves company entities across Japanese and English by normalizing, transliterating, and fuzzy-matching company names against a registry of ~50–100k Japanese corporate records sourced from the National Tax Agency (NTA). A query like “Sony” will match “ソニー株式会社” and vice versa.

Resolution Pipeline

Every query passes through six stages:

  • Language Detection — CJK Unicode range detection with langdetect fallback. Any hiragana/katakana triggers Japanese path.
  • Text Normalization — NFKC Unicode normalization, corporate suffix stripping (株式会社, Corporation, Ltd., etc.), case folding, whitespace collapse.
  • Transliteration — Japanese text is romanized via pykakasi (Hepburn). Long-vowel variants are generated (sonii → sony, soni).
  • Phonetic Encoding — A Soundex-inspired key groups phonetically similar consonants. “sony” and “soni” produce the same key.
  • Candidate Blocking — Trigram index + phonetic key lookup narrows ~100k records to ~200 candidates before expensive scoring.
  • Ensemble Scoring — Four weighted strategies score each candidate. The best score across all form pairings wins.

Matching Strategies

  • Jaro-Winkler (30%) — Prefix-weighted edit similarity. Good for typos and short strings.
  • Levenshtein (25%) — Normalized edit distance. Penalizes insertions, deletions, substitutions.
  • Token Sort (25%) — Sorts tokens alphabetically before comparing. Handles word reordering.
  • Phonetic (20%) — Compares Soundex-style keys. Catches cross-language matches that sound alike.

Data Source

  • Japan National Tax Agency (NTA) corporate number registry
  • ~50–100k records per prefecture (default: Kanagawa, code 14)
  • Fields: corporate number, company name (kanji), furigana, English name, address
  • Pre-computed columns: normalized name, romaji, phonetic key, character trigrams

Multi-Form Comparison

Each query generates multiple representations (original, normalized, romaji, phonetic key). Each candidate also has multiple forms. The scorer tries all compatible pairings and keeps the best. This is how “ソニー” matches “Sony” — through the romaji form “sonii”.

Technology

  • Python 3.11+, FastAPI, uvicorn
  • rapidfuzz (fuzzy matching), pykakasi (JP transliteration)
  • aiosqlite (async SQLite, ANSI SQL for Snowflake portability)
  • Pluggable strategy architecture via ABC registry
API Reference
REST endpoints available at this host. Full interactive documentation at /docs (Swagger UI).
MethodEndpointDescription
POST /search Search the corporate registry by company name. Returns ranked matches with scores and pipeline explanation. Body: {"query": "Sony", "limit": 10}
POST /match Compare two names directly without database lookup. Returns similarity score and per-strategy breakdown. Body: {"name_a": "ソニー", "name_b": "Sony"}
POST /batch Submit a batch of search queries for async processing. Returns a job ID for polling. Body: {"queries": [{"query": "Sony", "limit": 5}]}
GET /batch/{job_id} Poll batch job status and retrieve results when complete.
GET /health Liveness check. Returns {"status": "healthy"}.
GET /stats Database statistics: total companies and trigram count.
